`coding-prd-clarify` skill 提供交互式、半自动化的需求澄清流程，与 `coding-prd-analyzer` 无缝集成。通过 `/prd-clarify` 命令，用户可以提交澄清意见，系统自动关联到功能点、显示原型链接、引导用户验证，并生成时间戳化的澄清文件，保持原始文件不变。

---

## 输入格式

`/prd-clarify n=<requirement-name> "<user-opinion>" [options]`

参数说明：
- `n=<requirement-name>`: **必需**，需求名（与 `coding-prd-analyzer` 使用的需求名一致）。支持中文字符，特殊字符将自动规范化。
- `"<user-opinion>"`: **必需**，用户澄清意见，用双引号包裹。支持多意见，用分号（`;`）分隔。
- `-b=<filename>`: **可选**，指定基于哪个文件进行澄清（默认为最新澄清文件或原始 `analysis.md`）。
- `--history`: **可选**，显示该需求的澄清历史。

### 参数位置
- `n=` 参数必须在意见之前。
- 意见必须用双引号包裹，可以包含空格和标点。
- 选项可以在意见之前或之后。

示例：
- `/prd-clarify n=167023 "发起人权限应该支持多级审批"`
- `/prd-clarify n=用户登录 "登录页面需要添加忘记密码链接；注册流程需要手机号验证"`
- `/prd-clarify n=167023 -b=20260427-173000-analysis.md "发起人权限应该支持多级审批"`

## 需求名规范化与验证

需求名验证和规范化规则与 `coding-prd-analyzer` 保持一致：

### 验证规则
- **空值处理**: `n=` 后无内容将报错。
- **长度限制**: 需求名超过 100 字符将被截断。
- **危险字符过滤**: 文件系统危险字符（`../`, `*`, `?` 等）将被移除或替换。

### 规范化规则
- 保留中文字符、字母、数字、连字符、下划线。
- 将其他特殊字符（空格、斜杠、点号等）转换为下划线。
- 示例转换：
  - `用户登录 v1.0` → `用户登录_v1_0`
  - `订单/支付流程` → `订单_支付流程`
  - `user-auth-v2` → `user-auth-v2`（保持不变）

规范化后的需求名用于查找 `coding/prd/{requirement-name}/` 目录。

## 澄清流程

### 1. 查找最新版本目录
- 在 `coding/prd/{requirement-name}/` 下查找按时间戳排序的最新目录（`{timestamp}-prd-analysis`）。
- 如果目录不存在，报错并提示用户先运行 `coding-prd-analyzer` 生成分析文件。

### 2. 确定基础文件
- 检查版本目录下的 `.latest-clarification` 文件：
  - 如果存在，读取其内容，将其指向的文件作为基础文件。
  - 如果不存在，使用 `analysis.md` 作为基础文件。
- 如果指定了 `-b=<filename>` 选项，则使用该文件作为基础文件（需验证文件存在）。

### 3. 意见解析与关联
- 解析用户意见：按分号分隔多个意见。
- 对每个意见，提取关键词（中文分词或短语匹配）。
- 在基础文件中搜索功能点（`### 功能点 [number]: [title]`），匹配标题和描述中的关键词。
- 显示关联结果：每个意见关联到的功能点列表。

### 4. 原型验证辅助
- 从基础文件的 frontmatter 中提取 `prd_sources` 原型链接。
- 按类型（Axure、蓝湖、墨刀、飞书等）分类显示链接。
- 对于关联的功能点，高亮最相关的原型链接（基于功能点描述中的关键词匹配）。
- 生成验证提示：例如“请查看原型中发起人权限配置部分，确认是否需要支持多级审批？”

### 5. 交互确认
- 使用 `AskUserQuestion` 工具提供三个选项：
  1. **更新到新文件**：创建新的澄清文件，将澄清内容整合到功能点描述中。
  2. **仅保存记录**：在 `files/` 子目录下创建澄清笔记，不修改功能点文件。
  3. **取消**：放弃本次澄清。

### 6. 澄清文件生成（如果选择“更新到新文件”）
- 生成时间戳化文件名：`{YYYYMMDD-HHMMSS}-analysis.md`（例如 `20260427-173000-analysis.md`）。
- 复制基础文件内容。
- 根据用户意见修改关联的功能点描述（在描述末尾添加澄清说明，或更新现有描述）。
- 更新 frontmatter，添加澄清元数据：
  ```yaml
  based_on: "20260427-161029-analysis.md"   # 基础文件名
  clarification_summary: "发起人权限添加多级审批"  # 澄清摘要
  clarified_by: "user"                      # 澄清人（当前用户）
  clarified_at: "2026-04-27T17:30:00"      # 澄清时间（ISO 8601）
  ```
- 写入新文件。
- 更新 `.latest-clarification` 文件，使其指向新文件。

### 7. 澄清记录（如果选择“仅保存记录”）
- 在 `files/clarification-notes/` 子目录下创建时间戳化笔记文件。
- 记录意见、关联的功能点、验证结果、用户决定等。
- 不修改功能点文件。

## 文件组织

澄清文件与原始分析文件位于同一版本目录下，形成线性版本链。

```
coding/prd/167023/20260427-161029-prd-analysis/
├── analysis.md                    # 原始分析文件（保持不变）
├── 20260427-173000-analysis.md   # 第一次澄清文件
├── 20260427-180000-analysis.md   # 第二次澄清文件
├── files/                         # 中间文件和澄清记录
│   ├── clarification-notes/
│   │   ├── 20260427-173000-note.md
│   │   └── 20260427-180000-note.md
│   └── screenshots/              # 可选截图保存目录
└── .latest-clarification         # 纯文本文件，内容为最新澄清文件名
```

### 版本链管理
- `.latest-clarification` 文件存储最新澄清文件名（如 `20260427-180000-analysis.md`）。
- 每次创建新澄清文件后更新该文件。
- 如果文件不存在，则默认基于 `analysis.md`。
- 支持基于任意版本澄清（通过 `-b` 参数）。

### 澄清文件格式
澄清文件保持与原始 `analysis.md` 相同的 YAML frontmatter 和 Markdown 结构，仅增加澄清相关字段。功能点描述在原有基础上追加或更新，保留原始内容以追踪变更。

示例 frontmatter 新增字段：
```yaml
based_on: "20260427-161029-analysis.md"
clarification_summary: "发起人权限添加多级审批；参与人身份区分"
clarified_by: "user"
clarified_at: "2026-04-27T17:30:00"
```

## 半自动辅助验证

### 意见-功能点关联算法
- 关键词提取：使用简单的中文分词或短语匹配（如“发起人权限”、“多级审批”）。
- 匹配范围：功能点标题、描述、交互步骤、输入数据等字段。
- 关联评分：基于匹配关键词数量和位置计算简单评分，显示前几个关联结果。

### 手动关联支持
- 如果自动关联失败或用户不满意，用户可以手动指定功能点编号。
- 通过交互式提示让用户输入功能点编号（如“1,3,5”）。

### 原型链接提取与分类
- 从 `prd_sources` 字段提取所有链接。
- 按域名识别类型：`lanhu.app`（蓝湖）、`modao.cc`（墨刀）、`axure`（Axure）、`feishu.cn`（飞书）等。
- 分类显示，每个链接可点击（在支持的环境中）或复制。

### 验证提示生成
- 基于关联的功能点描述生成上下文相关的验证问题。
- 示例：“功能点 3 描述为‘发起人配置权限’，请查看原型中相关部分，确认是否需要支持多级审批？”

### 验证结果记录
- 用户验证后，记录结果：`confirmed`（确认合理）、`rejected`（不合理）、`needs_further_review`（需进一步讨论）。
- 结果保存在澄清笔记中。

## 错误处理

### 需求名不存在
- 如果 `coding/prd/{requirement-name}/` 目录不存在，报错：“未找到需求 '{requirement-name}' 的分析目录，请先使用 `/prd` 生成分析文件。”

### 版本目录不存在
- 如果需求目录下没有任何 `{timestamp}-prd-analysis` 目录，报错：“需求 '{requirement-name}' 下未找到分析版本，请先使用 `/prd` 生成分析文件。”

### 基础文件不存在
- 如果 `.latest-clarification` 指向的文件或 `analysis.md` 不存在，报错：“基础文件 '{filename}' 不存在，请检查版本目录。”

### 意见为空
- 如果用户意见为空或只包含空格，报错：“澄清意见不能为空。”

### 原型链接不可访问
- 提示用户手动验证，并提供原链接供参考。

## 使用示例

### 示例 1：基本澄清
```
/prd-clarify n=167023 "发起人权限应该支持多级审批"
```
流程：
1. 找到 `coding/prd/167023/` 下最新版本目录。
2. 读取基础文件（`.latest-clarification` 或 `analysis.md`）。
3. 关联意见到功能点（例如功能点 3）。
4. 显示原型链接（蓝湖链接）。
5. 提示用户验证。
6. 用户确认后，提供三个选项。
7. 若选择“更新到新文件”，生成 `20260427-173000-analysis.md`。

### 示例 2：多意见澄清
```
/prd-clarify n=用户登录 "登录页面需要添加忘记密码链接；注册流程需要手机号验证"
```
- 意见1：“登录页面需要添加忘记密码链接”
- 意见2：“注册流程需要手机号验证”
- 分别关联到功能点，独立处理。

### 示例 3：基于特定版本澄清
```
/prd-clarify n=167023 -b=0260427-173000-analysis.md "发起人权限应该支持多级审批"
```
- 基于 `20260427-173000-analysis.md` 文件进行澄清，而非最新文件。

### 示例 4：查看澄清历史
```
/prd-clarify n=167023 --history
```
- 显示该需求下所有澄清文件（按时间排序）及其摘要。

### 示例 5：仅保存记录
```
/prd-clarify n=167023 "发起人权限应该支持多级审批"
```
- 在交互确认步骤中选择“仅保存记录”，在 `files/clarification-notes/` 下创建笔记文件。

## 与 coding-prd-analyzer 集成

- 复用 `coding-prd-analyzer` 的需求名查找和原型链接提取逻辑。
- 依赖 `coding-prd-analyzer` 生成的目录结构和文件格式。
- 不修改 `coding-prd-analyzer` 的输出，确保向后兼容。

---

## 实现说明

- 使用 `AskUserQuestion` 工具提供交互选项。
- 文件操作使用 `Read`、`Write`、`Edit`、`Glob` 工具。
- 时间戳生成使用 `date +%Y%m%d-%H%M%S`。
- 保持代码模块化，便于扩展和维护。

---

## 向后兼容性

- 不修改现有 `coding-prd-analyzer` 生成的任何文件。
- 新澄清文件与原始文件格式一致，可被其他工具读取。
- 如果 `.latest-clarification` 文件不存在，回退到 `analysis.md`，确保首次澄清正常进行。

---

## 注意事项

- 澄清文件仅追加或更新功能点描述，不删除原始内容。
- 原型验证需要用户手动查看，skill 不进行自动视觉分析。
- 版本链为线性结构，不支持分支合并。
- 并发澄清可能导致 `.latest-clarification` 冲突，采用“最后写入胜出”策略。
