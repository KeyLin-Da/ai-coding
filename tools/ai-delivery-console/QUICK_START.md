# 🚀 AI 需求交付控制台 - 快速入门指南

## 5 分钟上手教程

### 第一步：启动服务

```bash
cd tools/ai-delivery-console
npm install
npm run server:dev  # 启动后端服务（端口 8718）
npm run dev         # 启动前端服务（端口 5178）
```

访问 http://127.0.0.1:5178 即可看到主界面。

---

### 第二步：创建第一个需求

1. **进入需求列表页面**
   - 首页会展示所有已存在的需求卡片
   - 每个卡片显示需求号、工作流状态和进度条

2. **点击「新建需求」按钮**
   - 输入需求编号（例如 `172014`）
   - 系统会自动创建工作流目录结构

3. **开始 PRD 分析**
   - 点击需求卡片进入详情页
   - 在左侧导航栏选择「PRD 分析」阶段
   - 点击「生成 PRD」按钮，或手动执行：
     ```
     /coding-prd-analyzer d=需求描述 r=172014
     ```

---

### 第三步：理解工作流阶段

AI 需求交付控制台将需求交付分为 5 个核心阶段：

#### 📝 阶段 1：PRD 分析
- **目标**：生成清晰的产品需求文档
- **产物**：`docs/172014/prd/analysis.md`
- **操作**：编辑 Markdown、调用 Agent 生成、添加澄清描述

#### 🏗️ 阶段 2：技术方案
- **目标**：设计技术实现方案
- **产物**：`docs/172014/technical-design/design_review.md`
- **操作**：查询数据库结构、绘制 Mermaid 图表、提交评审意见

#### ✅ 阶段 3：实施验证
- **目标**：通过 OpenSpec 规范和单元测试验证实现
- **产物**：`openspec/changes/req-172014/` + `docs/172014/junit/report.md`
- **操作**：查看任务清单、运行单元测试、检查覆盖率

#### 👨‍💻 阶段 4：代码评审
- **目标**：自动化代码质量审查
- **产物**：`docs/code_review/code_review_feature_opp_172014/summary.md`
- **操作**：选择分支、触发增量评审、查看问题分类

#### 📊 阶段 5：完成归档
- **目标**：确认所有产物就绪
- **操作**：检查工作流状态、导出完整报告

---

### 第四步：使用 Agent Provider

#### 模式 1：Codex CLI 自动执行（推荐）

1. 确保已安装 Codex CLI
2. 在页面右上角选择 Agent Provider 为 `codex`
3. 点击「执行」按钮，系统会自动：
   - 生成 Prompt 文件
   - 调用 Codex CLI 执行
   - 实时展示终端输出（SSE）
   - 更新产物状态

#### 模式 2：手动执行

1. 选择 Agent Provider 为 `manual`
2. 点击「执行」按钮
3. 复制生成的命令文本，例如：
   ```
   /coding-design d=docs/172014/prd/analysis.md r=172014
   ```
4. 在你的 Agent 环境中执行该命令
5. 执行完成后，回到页面点击「刷新产物」

---

### 第五步：查看实时日志

在任何阶段，点击右侧的「运行日志」按钮：

- 📜 查看 SSE 实时输出的终端日志
- ⏱️ 显示每个步骤的执行时间
- 🎨 彩色高亮关键信息
- 🔍 支持搜索和过滤

## 💡 最佳实践

### ✅ 推荐做法

1. **按顺序执行阶段**
   - PRD → 技术方案 → 实施验证 → 代码评审
   - 每个阶段的产物是下一阶段的基础

2. **善用二次评审**
   - 技术方案阶段可以输入评审意见
   - 系统会重新生成改进后的方案

3. **定期刷新产物**
   - 如果外部修改了文件，点击「刷新产物」同步状态
   - 避免 hash 冲突导致的覆盖问题

4. **使用澄清描述**
   - 在 PRD 阶段填写澄清描述
   - 会让生成的 PRD 更贴合实际需求

### ❌ 避免事项

1. **不要同时操作同一需求**
   - 系统有锁机制，但仍建议单人单需求操作

2. **不要手动修改 workflow 目录**
   - `docs/{需求号}/workflow/` 由系统管理
   - 手动修改可能导致状态不一致

3. **不要忘记保存**
   - Markdown 编辑器需要手动保存（Ctrl/Cmd + S）
   - 保存时会进行 hash 校验

## 🎯 典型使用场景示例

### 场景 A：新功能开发（需求号 172014）

```bash
# 1. PRD 分析
/coding-prd-analyzer d="实现用户积分系统" r=172014

# 2. 技术方案设计
/coding-design d=docs/172014/prd/analysis.md r=172014

# 3. 实施验证（OpenSpec + 单元测试）
/openspec-new-change req-172014
/coding-junit r=172014

# 4. 代码评审
/coding-review b=feature_opp_172014
```

### 场景 B：遗留代码补充测试

```bash
# 直接跳到单元测试阶段
/coding-junit r=166502

# 查看测试报告
# 访问 docs/166502/junit/report.md
```

### 场景 C：代码评审专项

```bash
# 增量评审（只评审新提交的代码）
/coding-review b=feature_opp_171634

# 核对外部文档约束
/coding-review b=feature_opp_171634 d=https://example.com/api-spec.md
```

## 🔧 常见问题速查

| 问题 | 解决方案 |
|------|---------|
| 端口被占用 | 修改 `vite.config.ts` 中的端口配置 |
| Agent 执行失败 | 检查 `CODEX_COMMAND` 环境变量 |
| 产物未更新 | 点击「刷新产物」按钮 |
| Hash 冲突 | 先刷新产物，再编辑保存 |
| 锁文件残留 | 删除 `docs/{需求号}/workflow/.lock` 文件 |
